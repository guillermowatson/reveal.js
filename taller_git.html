¿Qué es un sistema de control de versiones?

Se ocupa de registrar los cambios en un archivo o serie de archivos a lo largo del tiempo para poder recuperar cualquier versión anterior.

SCM (Source Code Managment)
SVC (System Version Control).
CVS, Subversion, Bazaar, Mercurial y otros...
Diseñador gráfico o web. Programador o Escritor...

Si alguna vez te pasó esto

(a mí me pasó más de una vez...)

Tener varias versiones en diferentes carpetas sin saber bien cuál es la correcta.

Sería buenísimo implementar un sistema de control de versiones

Git te puede ayudar a no tener más esos problemas.

¿Por qué Git?
• Es rápido, es confiable, fácil de usar.
• Las versiones del kernel de linux se controlan con Git.
• Desarrollos no lineales --> soporta ramas.
• Creado por Linus Torvalds en el 2005
• Escrito en C, Bourne Shell, TCL y Perl
• Licencia GPL v.2

Sistema Local de Control de Versiones

"Copiar los archivos en otro directorio"


es fácil equivocarse y sobreescribir por error...
Hacer las copias en otros directorios es propenso a errores.
Se desarrollaron VCS locales con una simple DB.

2 Modelos

Centralizado vs. Distribuido

Centralizado:
• Todo el mundo puede saber (hasta cierto punto) en qué están trabajando los otros colaboradores.
• Más fácil administrar un CVCS que tener que lidiar con DB locales.
• FALLO SERVIDOR ---> WARNING!
Distribuido:
• Cada cliente tiene una copia completa.
• MERGE --> con cuidado.

Instantáneas, no diferencias

Git modela los datos como un conjunto de "fotos" de un mini file-system.

Muchos sistemas sólo guardan las diferencias, como un BackUp Incremental.
Git utiliza enlaces para ser más eficiente.

Características

  1. Casi todas las operaciones son locales.
  2. Mantiene la integridad por medio de cheksums.
  3. Generalmente sólo añade información.
FRAGMENTS:
• LOCAL: Todo ocurre en tu máquina.
• ÍNTEGRO: Se utilizan HASH (20 bytes - 40 dígitos) para verificar la integridad de los archivos (SHA-1).
• INCREMENTAL: El sistema es incremental.

Los 3 Estados

  1. Trabajo.
  2. Preparación.
  3. Confirmado.

FRAGMENTS:
1) Trabajo: nuestros archivos no modificados.
2) Preparación: nuestros archivos modificados listos para ser registrados.
3) Confirmado: nuestros archivos ya registrados en el sistema.

entonces...

Configurando Git

Git viene con una herramienta que se llama: git config


Sistema

/etc/gitconfig


Global

~/.gitconfig


Repositorio

.git/config


Las opciones puestas en el repositorio sobreescriben la global y la del sistema.
Las opciones puestas de manera global (por usuario) sobreescriben las del sistema.
Las opciones del sistema son utilizadas si no existen opciones globales ni del repositorio.

Creando un repositorio

En un nuevo directorio iniciamos un repositorio

Abrimos una terminal y ejecutamos lo siguiente:

 
~ $ mkdir Manual_Programacion
~ $ cd Manual_Programacion
~/Manual_Programacion $ git init
Se puede descargar los archivos de bitson.com.ar/flisol/archivos.tgz

Agregando archivos

Ya tenemos listo nuestro repositorio. Git "le presta atención" a lo que hagamos dentro de este directorio.

Colocamos los archivos que van a formar parte de nuestro repositorio en el directorio.

Y debemos decirle a Git que queremos agregar estos archivos al repositorio.

 
   
~/Manual_Programacion $ git add *
~/Manual_Programacion $ git status
~/Manual_Programacion $ git commit -m "Iniciando el repositorio"
Podemos agregar archivos por su nombre o con expresiones.
AGREGAMOS EL CONTENIDO DE LA CARPETA "01" (2 archivos: Banner.png y Manual.odt)
Después de copiar, podemos tirar un status para ver lo que Git detecta.
COMMIT --> STATUS

Workflow

Los archivos en un directorio de trabajo pueden estar: • BAJO SEGUIMIENTO o
• SIN SEGUIMIENTO.
Si están BAJO SEGUIMIENTO:
• Sin modificaciones,
• modificado y
• preparados.

¿En qué estado estamos?

 
~/Manual_Programacion $ git status
On branch master
nothing to commit, working directory clean
Branch = rama.
Por ahora siempre será MASTER (principal).

Agregando más archivos

Agreguemos un simple archivo README.

Observemos cómo Git nos informa sobre el estado.

 
~/Manual_Programacion $ vim README
~/Manual_Programacion $ git status
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)

       README

nothing added to commit but untracked files present (use "git add" to track)
AGREGAMOS EL CONTENIDO DE LA CARPETA "02" (archivo README)
El archivo README no está siendo seguido por Git.
• UTRACKED (sin seguimiento) => hay que agregarlo con ADD.

Siguiendo nuestros archivos

 
~/Manual_Programacion $ git add README
~/Manual_Programacion $ git status
On branch master
Changes to be committed:
  (use "git reset HEAD <file>..." to unstage)

     new file:   README
Ahora README está siendo seguido por Git. "STAGED" = to be commited.

Preparando archivos modificados

Hagamos una pequeña modificación en el archivo: manual.odt.

Agregamos los cambios

Si queremos agregar todos los cambios en todos los archivos:

  (use "git reset HEAD <file>..." to unstage)

Recursivamente.

Registrando cambios

Entonces ahora sólo resta registrar estos últimos cambios.

 
~/Manual_Programacion $ git commit

Dentro del COMMIT:

Agregando archivo README y modificación de manual.

Hasta ahora iniciamos el repositorio, y después agregamos un archivo nuevo
(README), hicimos una modificación en el manual y ahora estamos fijando el
estado para que quede una versión.
Se abre el editor de texto que configuramos y describimos el cambio.
En la primera línea una breve descripción.
Dejamos una línea en blanco y podemos explayarnos todo lo que querramos.

¿Tenemos que agregar TODOS los archivos del directorio de trabajo?

Ignorando algunos archivos

Crear en el directorio raíz de nuestro repositorio el archivo: .gitignore

  • Las líneas en blanco o que empiezan por # son omitidas.
  • Se pueden usar patrones glob estándar.
  • Se puede indicar un directorio agregando una barra (/) al final del nombre.
  • Se puede negar un patrón añadiendo ! adelante del patrón.
Los patrones glob son expresiones regulares simplificadas que pueden ser usadas por las shells.
Por ejemplo, archivos temporales. Abrir manual.odt y tirar un STATUS aparece un archivo ".~lock.manual.odt#"

Por Ejemplo

Creamos el archivo .gitignore y agregamos:

###########################################################################
## Acá agregamos los patrones que queremos ignorar.
##
##  - Las líneas en blanco o que comienzan con '#' son ignoradas
##  - Se pueden usar patrones GLOB estándar:
##      - *: Un asterisco reconoce cero o más caracteres.
##      - [xyz]: Cualquier conjunto de caracteres, en este caso, x o y o z.
##      - ?: Un signo de interrogación reconoce un único caracter.
##      - [a-d]: Caracteres entre corchetes separados por un guión reconoce
##               cualquier caracter entre ellos. En este caso: a, b, c o d.
##  - Se pueden indicar directorios con una barra al final. Por ejemplo: 
##      Temporales/
##  - Se puede negar cualquier patrón agregando un signo de admiración (!) 
##    adelante del patrón. Por ejemplo:
##      !*.c
###########################################################################

## Ignorando archivos temporales de libreoffice:
.~*
Con esto le estamos indicando que debemos omitir todos los archivos que comiencen con un punto y seguido de '~'.
AGREGAMOS EL CONTENIDO DE LA CARPETA "04" (archivo gitignore: COPIARLO COMO OCULTO!!!!).
git status detecta la inclusión del gitignore. => ADD!

Agregando al repositorio

Ahora debemos agregar el .gitignore al repositorio:

~/Manual_Programacion $ git add .gitignore

Abrir el manual.odt y consultar el estado del repositorio:

 
~/Manual_Programacion $ git status

Notemos como nos aparece el archivo temporal.

Antes de hacer el COMMIT, abrir el archivo manual y tirar un STATUS.
FRAGMENTS: Aparece el archivo temporal
En la proxima diapo, después del COMMIT veremos los cambios!

Saltando 1 paso

Se puede omitir el estado de preparación (staged) utilizando la opción -a.

 
~/Manual_Programacion $ git commit -a -m "Agregando archivo .gitignore"

Volver a abrir el manual.odt y a consultar el estado del repositorio:

 
~/Manual_Programacion $ git status
El parámetro -a nos ahorra un ADD en caso de haber hecho modificaciones.
• FRAGMENT: Volver a abrir el manual y tirar un STATUS después de haber hecho el COMMIT.

Viendo los cambios

¿Cómo podemos hacer para visualizar los cambios realizados hasta el momemento?

 
~/Manual_Programacion $ git log
Vamos a ver:
• El HASH del commit,
• quién lo realizó,
• en qué momento
• y la descripción que ha hecho.
Con el parámetro --pretty=format:"". Consultar libro u hoja de manual para los formatos.

Eliminando archivos

Se deben eliminar del área de seguimiento (staged). Se utiliza el comando git rm que lo elimina del repositorio y del directorio de trabajo.

Copiar archivos dentro del directorio de trabajo:

 
~/Manual_Programacion $ git add .
~/Manual_Programacion $ git commit -a -m "Agregando material"

Ahora sí, después de haber registrado los cambios, eliminamos la carpeta:

 
~/Manual_Programacion $ git rm -r Material/

Verificamos el estado, hay que volver a registrar los cambios:

 
~/Manual_Programacion $ git commit -a -m "Sacando material"
Si sólo lo eliminás del directorio de trabajo con rm aparecerá como "Modificados pero no actualizados".
El comando git rm acepta patrones GLOB.
AGREGAMOS EL CONTENIDO DE LA CARPETA "05" (carpeta Material/).
FRAGMENTS:
• COMMIT "Agregando Material"
• Eliminar Carpeta
• COMMIT "Sacando Material"

Eliminando del seguimiento

Usamos el parámetro --cached .

Copiamos archivos al directorio de trabajo y los agregamos.

 
~/Manual_Programacion $ git status
~/Manual_Programacion $ git add .
~/Manual_Programacion $ git status

UPS! Fue un error... entonces los borramos del seguimiento:

 
~/Manual_Programacion $ git rm --cached -r Material/
~/Manual_Programacion $ git status

Debemos ignorarlos:

 
~/Manual_Programacion $ git commit -a -m "Omitiendo Material/ (.gitignore)"
Después de borrar con cached deberás incluir ese archivo al gitignore para que no se vuelva a agregar por error.
FRAGMENTS: AGREGAMOS EL CONTENIDO DE LA CARPETA "05" (carpeta Material/).
• Agregamos los archivos "por error".
• Eliminamos del seguimiento.
AGREGAMOS EL CONTENIDO DE LA CARPETA "06" (archivo gitignore actualizado!).
• Ignoramos los archivos.

Renombramos el archivo README a README.txt:

 
~/Manual_Programacion $ git mv README README.txt
~/Manual_Programacion $ git status

Confirmamos el cambio:

 
~/Manual_Programacion $ git commit -a -m "Renombrando README"
~/Manual_Programacion $ git status

Verificamos en el SO:

Deshacer modificaciones

Después de realizar un cambio, podemos volver a la versión anterior en la etapa staged.

~/Manual_Programacion $ git checkout -- <file>
Realizamos otro cambio en el manual.odt. Abrir y poner cualquier cosa.
STATUS --> Para ver el cambio detectado.
Si queremos descartar el cambio recién realizado:
git checkout -- manual.odt
Si abrimos el manual.odt, veremos que ha vuelto al estado original! SOBREESCRITO!
STATUS --> Estamos sin cambios ;-)

Volver sobre nuestros pasos

Si fuese necesario volver a algún commit anterior porque los cambios realizados no nos convencen.

Utilizaremos el comando git reset. Para esto necesitamos saber el hash del commit, ya que así los identificaremos.

Buscando commits

Utilizando el comando git log y el comando git reflog, podremos ver a qué commit nos queremos mover.

 
~/Manual_Programacion $ git log --pretty=format:"%h %s" 
                        
 
~/Manual_Programacion $ git reflog
Cada uno de ustedes va a tener seguramente un hash diferente.

Pasos firmes o ...

Para hacer un cambio "suave":

~/Manual_Programacion $ git reset --soft <commit>

O un cambio "fuerte":

~/Manual_Programacion $ git reset --hard <commit>
SOFT no toca el índice. Vuelve sólo el puntero HEAD al COMMIT indicado.
HARD, descartará todos los cambios y volverá al COMMIT indicado.

Volvemos al 1º COMMIT con soft.
STATUS --> DETECTA TODO LO "NUEVO" QUE HAY.
Volver al último COMMIT con soft.

Repetir con hard. Mirar el directorio con "ls".

Ramificándonos

¿Qué es una rama?

Maneja las ramificaciones es increíblemente rápida.
Promueve un ciclo de desarrollo donde las ramas se crean y se unen.
Poderosa y exclusiva herramienta.
Una rama es realmente un simple archivo que contiene los 40 caracteres SHA-1
FRAGMENTS: • IMAGEN RAMAS MASTER.

Creando una rama

Creamos una rama llamada testing.

 
~/Manual_Programacion $ git branch testing

¿Cómo sabe Git dónde estamos trabajando?

Saltando de Rama en Rama

Cambiamos el puntero HEAD con el comando checkout.

 
~/Manual_Programacion $ git checkout testing
git branch --> Informa en qué rama estás.

Haciendo cambios en la nueva rama

Editemos el archivo README.

 
~/Manual_Programacion $ git commit -a -m "Editando README en rama TESTING."
AGREGAMOS EL CONTENIDO DE LA CARPETA "11" (archivo README MODIFICADO!).

Volviendo a Master

 
~/Manual_Programacion $ git checkout master

Modificando Master

Editemos el archivo README.

 
~/Manual_Programacion $ git commit -a -m "Editando README en rama MASTER."
AGREGAMOS EL CONTENIDO DE LA CARPETA "12" (archivo README MODIFICADO!).

Divergente

¿qué hemos hecho?

Saltar de rama en rama con CHECKOUT y ver el contenido del README

Mezclando

Para fusionar nuestrar ramas tendremos que utilizar el comando merge.

 
~/Manual_Programacion $ git merge testing

Si no necesitamos más la rama la podemos borrar con el parámetro -d.

 
~/Manual_Programacion $ git branch -d testing
Con git branch muestra en qué rama estamos parados.

Resolviendo conflictos

A veces Git no puede resolver solo la unión de las ramas y necesitará que decidamos qué hacer.

Con status nos informará qué no pudo resolver.

Editemos (resolviendo) el archivo y agreguemos a la etapa staged.

 
~/Manual_Programacion $ vim README.txt
~/Manual_Programacion $ git add README.txt

Terminemos la fusión con commit.

 
~/Manual_Programacion $ git commit
Con STATUS nos informa que tenemos que resolver un conflicto con README.txt.
Unmerged => Sin Fusionar.
Abramos el archivo README y veamos el contenido.
COMMIT abre el editor de texto y confirmamos la fusión con la descripción.

  2. Crear un repositorio, presionando sobre el botón Crear.
  3. Completar el formulario. Terminamos con el botón Crear Repositorio.
https://bitbucket.org/<username>/<repositorio>
FRAGMENTS:
• Crear cuenta
• Crear repositorio
• Formulario
• URL

Subiendo el repositorio

~/Manual_Programacion $ git remote add <nombre> https://<username>@bitbucket.org/<username>/<repositorio>.git
